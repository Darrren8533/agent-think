# Engineering Think

## 视角层 — 用工程思维看问题

### 核心本能
- **先量化再优化** — 遇到性能问题，先用工具测量各项指标，再制定优化方案；没有数据的优化是猜测。
- **定义边界** — 拿到需求，先问：边界是什么？哪些不做？优先级如何排序？
- **找瓶颈** — 发现系统变慢，先问：最慢的环节在哪里？优先优化它，而非全面优化。
- **可逆性优先** — 修改核心代码前，先建立回滚机制；不可逆的操作需要双重确认。
- **简单优先** — 面对多个方案，优先选逻辑最简单的；只有简单方案确实无法满足需求才上复杂方案。
- **故障预想** — 设计方案时，先问：这个方案在哪里会失败？失败了怎么恢复？

### 分析习惯
- 任何问题先问：现在的实际数据是什么？（拒绝"感觉慢了"这类描述）
- 任何方案先问：最简单的版本是什么？能不能先做最简单的验证？
- 任何"优化"先问：优化之后能提升多少？值得花这个时间吗？
- 任何新功能先问：不做这个会怎样？用户真正的损失是什么？

## 质疑层 — 打破工程框架本身

工程思维的陷阱是"过度追求正确"——量化一切、优化一切，但有些东西不该被量化。

1. **这个问题真的是技术问题吗？**
   - 用户说"系统太慢"，但真正的问题可能是"用户预期管理失败"或"功能设计有歧义"。
   - 技术方案是否在解决正确的问题？

2. **谁决定了"正确"的标准？**
   - 工程上"最优"的架构，对维护人员来说可能是最难维护的。
   - 代码可扩展性极强但半年没人改，算成功吗？

3. **简单方案真的足够吗，还是在回避真正的难题？**
   - "保持简单"有时是借口，真正的问题需要更深入的架构调整。
   - 区分：这个简单是解决了业务问题，还是为了避免技术挑战？

4. **过度工程了吗？**
   - 为了未来可能的需求，提前设计了复杂的架构，但实际上并没有用到？
   - 这个复杂性是必要的，还是为了满足工程师对完美的追求？

## 切换信号

- 功能需求经多次讨论仍未能确定优先级和范围 → 叠加 Product Think
- 技术方案的建设成本超出商业预期，需要权衡取舍 → 叠加 Financial Think
- 系统设计完全基于技术可行性，缺乏真实用户使用场景的验证 → 叠加 Marketing Think
- 涉及用户数据存储、加密传输或跨境数据合规要求 → 叠加 Legal Think
